SimpleCMP — Live-Demos

Sieben Demos, die Phase 1–4 und den Lit-UI-Rewrite in Aktion zeigen. Die Bundles werden direkt aus ../dist/ geladen — wenn du gerade Code änderst, läuft pnpm build davor.

1. Banner + Compliance-Basics

Klaro-Banner, getrennte Datenschutz/Impressum-Links (REQ-1), gleichberechtigte Decline/Accept-Buttons (REQ-2), Floating-Trigger zum Wieder-Öffnen (REQ-4), Modal-Accessibility (REQ-6).

2. Recorder mit Live-Panel

Wie #1, plus aktiver Recorder. Buttons unten lösen Cookie-/Script-/Iframe-Detections aus, ein Live-Panel zeigt was der Recorder findet. Inklusive exportConfig()-Snippet-Generator und assertNoUnknown()-Test.

3. Full-Stack mit Service-DB

Wie #2, plus DB-Lookup gegen die ddev-Reference-Backend. Detections werden mit Vendor-Metadata angereichert. Health-Check-Button gegen /v1/health.

⚠️ Erfordert vorab gestartetes Backend: cd reference-server && ddev start

4. Lit-UI im Detail

Wie #1, aber mit zusätzlichen Hooks: SimpleCMP.init() gibt ein Handle zurück (show(), hide(), destroy(), manager) — siehe REQ-14 / ADR-0007.

5. Lit-UI mit Bootstrap-Theme

Wie #4, plus dem Bootstrap-Adapter-Stylesheet (simplecmp/styles/bootstrap.css). Eine einzige <link>-Zeile mappt SimpleCMPs Custom Properties auf --bs-* — kein JS, kein Fork.

6. Light-DOM-Modus mit default.css

Wie #4, aber alle Components mit domMode: 'light'. Statt Shadow-DOM-Encapsulation lädt der Host simplecmp/styles/default.css direkt — host-CSS schlägt durch, Custom Properties cascaden ungehindert (REQ-16, ADR-0007).

7. CMS Bridge (Phase 4)

Wie #2, plus Webhook-Bridge. Unbekannte Detections lösen einen fetch-POST an einen lokalen Receiver (cms-bridge-receiver.mjs, Port 8787) aus — Live-Panel zeigt das empfangene Payload, Dedup-TTL ist demo-friendly auf 30 s gesetzt (REQ-9).

ⓘ Receiver wird von pnpm demo automatisch mitgestartet.
Alle Demos brauchen einen frischen Build. Aus dem Repo-Root: pnpm build (oder pnpm demo, das tut beides).
